
Note on updateNodeData with full replacement:
When json-edit-react calls setData, it provides the COMPLETE updated step object.
Passing the full object to updateNodeData's spread merge (`...step, ...patch`) is safe
because the full object overwrites all keys. No separate "replaceNodeStep" action needed.

1. Install json-edit-react:

2. Create `src/components/panel/JsonFallbackEditor.tsx`:
   - Props: `{ nodeId: string; step: Record<string, unknown> }`
   - Get `updateNodeData` from `useAppStore`
   - Render `JsonEditor` from `json-edit-react` with:
     - `data={step}` -- the full step object
     - `setData={(newData) => updateNodeData(nodeId, newData as Record<string, unknown>)}`
       The setData prop receives the complete updated object from json-edit-react.
       Passing it to updateNodeData replaces the full step (spread merge with full data is equivalent to replacement).
     - `rootName={nodeId}` -- shows node ID as root label
     - `collapse={2}` -- collapse after 2 levels deep for readability
     - `minWidth="100%"` and `maxWidth="100%"` -- fill panel width
     - Do NOT import a named theme (e.g., githubDarkTheme). Use default styling for now.
       Dark mode theming will be addressed in Phase 7.
   - Wrap in a div with `border rounded-lg overflow-hidden` for visual containment

   IMPORTANT per research pitfall 1: json-edit-react manages its own internal state after
   receiving the initial `data` prop. The `setData` callback is how it notifies the parent.
   Do NOT force re-renders by deriving `data` from a rapidly-changing selector. Pass the step
   object and let the library handle internal edits. The key anti-pattern is using a store
   selector that creates a new reference on every render -- instead, the step object reference
   only changes when a different node is selected or when structured fields edit the same node.

3. Create Wave 0 test stub `src/components/panel/__tests__/JsonFallbackEditor.test.tsx` (required by VALIDATION.md for EDIT-04 coverage):

   - `describe('JsonFallbackEditor')` with tests:
     - `it('renders the JSON editor with step data')` -- render with a sample nodeId and step object, assert the mock JsonEditor received the correct data prop
     - `it('calls updateNodeData when setData is triggered')` -- simulate the mock calling its setData prop, verify store's updateNodeData was invoked with the correct nodeId and new data
   - These must be real working tests, not placeholder assertions. Set up the Zustand store with sample flow data before each test.
